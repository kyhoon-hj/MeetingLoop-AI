# MeetingLoop AI 회의록 시스템 작업 계획서

## 1. 문서 목적

이 문서는 MeetingLoop AI를 현재의 메모리 기반 데모에서 PostgreSQL 기반 서비스로 전환하고, 수정된 최종 전사 TXT와 회의록을 저장·검색·조회·수정할 수 있도록 구현하기 위한 우선순위별 작업 계획서다.

## 2. 확정된 데이터 저장 정책

### 서버에 저장하는 데이터

- 사용자, 조직, 조직별 권한
- 프로젝트와 회의 기본정보
- 참석자와 안건
- 사용자가 수정하고 확정한 최종 전사 TXT
- 최종 전사의 발화자, 순서, 시작·종료 시각
- AI가 생성하고 사용자가 수정한 회의록
- 회의록 상태, 작성자, 수정자, 작성·수정 시각
- 전사와 회의록의 수정 이력

### 서버에 저장하지 않는 데이터

- 원본 음성 파일
- 브라우저 녹음 청크
- 수정 전 임시 전사
- 브라우저에서 작성 중인 미확정 전사

원본 음성과 녹음 청크는 브라우저 로컬 파일 또는 IndexedDB에만 보관한다. 서버 API, DB, 로그 및 백업에는 원본 음성을 남기지 않는다.

### 기본 처리 흐름

```text
브라우저 녹음
→ 음성 및 임시 전사를 브라우저에 보관
→ 사용자가 전사를 수정
→ 최종 전사 확정 및 PostgreSQL 저장
→ 저장된 최종 전사로 AI 회의록 생성
→ 사용자가 회의록 수정 및 확정
→ 최종 회의록과 변경 이력을 PostgreSQL 저장
→ 목록, 검색, 조회 및 재수정
```

## 3. 목표 기능

- 컨테이너 재시작 후에도 데이터 유지
- DB 기반 로그인, 조직 및 프로젝트 관리
- 최종 전사 저장, 조회, 수정 및 TXT 다운로드
- AI 회의록 생성 및 확정 저장
- 회의록 목록과 상세 조회
- 제목, 요약, 회의록 및 최종 전사 통합 검색
- 프로젝트, 기간, 상태별 필터
- 회의록 및 전사 수정 이력
- 동시 수정 충돌 방지
- 조직별 데이터 격리 및 역할별 권한 제어

### 권장 전사 구조

```text
transcripts
├─ id
├─ organization_id
├─ meeting_id
├─ status                 # CONFIRMED, DELETED
├─ version
├─ confirmed_by
├─ confirmed_at
├─ created_at
└─ updated_at

transcript_segments
├─ id
├─ transcript_id
├─ sequence
├─ speaker_label
├─ start_ms
├─ end_ms
├─ edited_text
├─ status
├─ edited_by
├─ created_at
└─ updated_at
```

TXT 파일 자체를 중복 저장하기보다 세그먼트를 정규 데이터로 저장하고, 조회 또는 다운로드할 때 TXT를 생성한다. 필요하다면 확정 시점의 전체 TXT snapshot을 revision에 보관한다.

### 완료 조건

- 최종 전사와 회의록을 회의별로 한 건의 현재 버전과 여러 revision으로 관리할 수 있다.
- 조직이 다른 데이터는 쿼리 수준에서 조회되지 않는다.
- 한국어 부분 문자열 검색을 위한 index가 생성된다.

- [ ] DB pool, 지연시간 및 오류율 모니터링
- [ ] API rate limit
- [ ] 요청 body 크기 제한
- [ ] 민감한 전사 내용 로그 제외
- [ ] 데이터 보존 기간에 따른 자동 삭제 작업

### 완료 조건

- 백업에서 사용자, 최종 전사 및 회의록을 복원할 수 있다.
- migration 실패 시 기존 서비스가 손상되지 않는다.
- 주요 장애와 보안 시나리오가 자동 테스트로 검증된다.

## 6. 권장 개발 순서와 의존성

```text
단계 0 정책 확정
   ↓
단계 1 PostgreSQL/마이그레이션
   ↓
단계 2 스키마 보완
   ↓
단계 3 인증/기본 데이터 DB 전환
   ↓
단계 4 최종 전사 저장
   ↓
단계 5 AI 회의록 저장
   ↓
단계 6 목록/상세 화면
   ↓
단계 7 검색/필터
   ↓
단계 8 권한/감사
   ↓
단계 9 테스트/운영 안정화
```

단계 1~5를 완료하기 전에는 기존 메모리 저장소를 제거하지 않는다. DB 구현과 검증이 완료되면 환경변수 또는 repository 교체 지점을 통해 DB 저장소로 전환한 뒤 메모리 구현을 제거한다.

## 7. 우선 개발 마일스톤

### 마일스톤 A: 데이터 영속화 기반

결과:

- DB 로그인
- 조직, 프로젝트 및 회의 저장
- 재배포 후 기본 데이터 유지

결과:

- 저장된 전사 기반 AI 회의록 생성
- 최종 회의록 저장 및 수정 이력

### 마일스톤 C: 조회 및 검색 서비스

범위: 단계 6~8

- 최종 전사 및 회의록 통합 검색

### 마일스톤 D: 운영 준비 완료

- 운영 배포 체크리스트

## 8. 전체 완료 기준

1. 컨테이너를 재시작해도 사용자, 프로젝트, 회의, 최종 전사 및 회의록이 유지된다.
2. 원본 음성과 미확정 전사는 서버 DB, 파일시스템 및 로그에 남지 않는다.
3. 최종 전사 저장 후 AI 회의록을 생성할 수 있다.
4. 회의록을 수정하고 이전 revision을 조회할 수 있다.
5. 회의 제목, 회의록 및 최종 전사 내용으로 검색할 수 있다.
6. 프로젝트, 기간 및 상태 필터가 동작한다.
7. 두 사용자가 동시에 수정해도 기존 변경을 덮어쓰지 않는다.
8. 다른 조직 사용자는 회의 ID를 알아도 데이터를 조회할 수 없다.
9. DB 장애 시 readiness가 실패하고 서비스가 잘못된 성공 상태를 반환하지 않는다.
10. 백업에서 최종 전사와 회의록을 복구할 수 있다.

## 9. 첫 번째 작업 묶음

개발은 다음 작업부터 시작한다.

- [ ] PostgreSQL 운영 방식 및 접속 정보 확정
- [ ] 실제 migration runner 구현
- [ ] DB pool 및 transaction 구현
- [ ] 현재 스키마에 transcript header와 revision 구조 추가
- [ ] 사용자·조직·프로젝트 repository 구현
- [ ] DB 통합 테스트 환경 구성

첫 번째 작업 묶음이 완료되면 인증과 기본 데이터가 PostgreSQL에서 정상적으로 유지되는지 확인한 후 최종 전사 저장 기능으로 진행한다.
